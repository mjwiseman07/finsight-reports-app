## Prerequisite Finding — Customer Role Deployment State (Task 1)

### Question

Can the system currently determine **"is target role X deployed for this customer?"**

### Answer: GAP

There is **no queryable deployment state** and **no function** that answers `(customer, roleType) → deployed?`.

### What exists today (fragments, not a registry)

| Artifact | Module | Relevant fields | Limitation |
|----------|--------|-----------------|------------|
| `SyntheticRoleActivation` | `lib/intelligence/synthetic/roles/rapid-role-activation/buildRoleActivation.ts` | `roleType`, `roleInstanceId`, `companyId`, `firstTaskReady`, `activationCurrentStep` → `step_10_activation_complete` | Per-activation builder output; no customer-wide index |
| `SyntheticWorkforceMember` | `lib/intelligence/synthetic/organization/workforce-registry/buildWorkforceMember.ts` | `roleReference`, `memberStatus: "active" \| "inactive" \| "pending"`, `aiWorkerSourcePhase39CompositionId` | `roleReference` is opaque string; no typed lookup |
| `SyntheticDeploymentArtifact` | `lib/intelligence/synthetic/organization/marketplace/buildDeploymentArtifact.ts` | `deploymentStatus: "deployed"`, `targetOrganizationalUnitId` | Marketplace / OU deployment, not per-role-type per customer |
| `SyntheticDigitalDepartment` | `lib/intelligence/synthetic/organization/digital-departments/buildDigitalDepartment.ts` | `aiWorkerReferenceIds`, `aiWorkerPhase39CompositionReferenceIds` | Department membership; no `(roleType → deployed)` resolver |
| Role compositions | `lib/intelligence/synthetic/roles/ai-*-accountant/`, `ai-controller-helper/` | `roleActivationReferenceId`, `taskQueueReferenceId` | Reference IDs only; no live deployment query |

All of the above remain **`executable: false`** metadata builders.

### What does NOT exist

- No `isRoleDeployed(companyId, roleType)` (or equivalent)
- No central **Customer Role Deployment Index** keyed by isolation scope + `roleType`
- No verifier check for deployment-state consistency across activation + workforce registry
- Phase 40C Workforce Registry stores members but does not expose **deployment resolution** for ad-hoc routing

### Minimal state needed (design only — blocks advise-to-deploy)

**Proposed contract:** `CustomerRoleDeploymentRecord` (extends org base contract; `executable: false`)

| Field | Purpose |
|-------|---------|
| `companyId` + `customerIsolation` / `firmIsolation` / `clientIsolation` | Tenant scope |
| `roleType` | e.g. `senior_accountant`, `controller_helper` |
| `deploymentStatus` | `"not_deployed" \| "activating" \| "deployed" \| "deactivated"` |
| `roleInstanceId` | Active instance when deployed |
| `roleActivationReferenceId` | Link to Phase 39 activation artifact |
| `workforceMemberReferenceId` | Link to 40C workforce member |
| `taskQueueReferenceId` | Target queue for routed handoffs |
| `deploymentResolvedAt` | Snapshot timestamp |

**Proposed DEPLOYED predicate (fail-closed):**

`deploymentStatus === "deployed"` **AND** linked activation at `step_10_activation_complete` (or `firstTaskReady === true` with non-empty `activationCompletedAt`) **AND** workforce `memberStatus === "active"`.

**Proposed home:** **40C-EXT** submodule `organization/workforce-registry/deployment-index/`. Population when activation completes or marketplace deployment approves — **writers out of scope for Path A addendum**; index contract + read resolver in scope.

Until this exists, **advise-to-deploy cannot be implemented correctly**.

## Phase 40 Extension — Ad-Hoc Authority Routing (Path A)

### Problem statement

Phase 39 declares lanes (`allowedTaskFamilies` / `forbiddenTaskFamilies`), escalation targets, and governance gates — but nothing evaluates an ad-hoc task against the acting role or routes above-authority work. Phase 40E/40F produce **recommendations** only. This extension closes the **ad-hoc authority routing** gap while preserving Phase 40 non-execution guardrails.

### Behavior rules (binding)

#### Authority evaluator

Classifies each ad-hoc task as one of:

| Outcome | Rule (reads existing Phase 39 defs only) |
|---------|------------------------------------------|
| `forbidden` | Task family ∈ acting role's `forbiddenTaskFamilies` (39D) |
| `in_lane` | Task family ∈ `allowedTaskFamilies` AND capability `reviewLevel` does not exceed acting role tier AND no fraud/reasonableness flag |
| `above_authority` | Task family allowed but `reviewLevel` (39 capability) exceeds acting role OR task family requires higher tier per capability map |
| `requires_human` | Fraud or reasonableness **flagged** (hard stop — see below) |

**Explicit non-goals for evaluator:** no new materiality engine; no new authority tiers; no `executable: true`.

**Inputs:** work-item descriptor (`taskFamily`, optional capability ref), acting `roleType` + `roleInstanceId`, loaded `SyntheticRoleRestriction`, matching `SyntheticRoleCapability`(ies), optional JE fraud/reasonableness status.

**Outputs:** `AuthorityEvaluationResult` contract (`authorityOutcome`, `escalationTargetRoleType`, booleans for email-intake hooks, `approvalRoutingTargetRoleType` from 39E governance).

#### Routing by outcome

**`in_lane` (Path A)**

- Task remains in acting role's queue (`role-task-queue` artifact updated — assignment metadata only).
- Acting role **prepares** output per existing 39E gates.
- Human gates unchanged (`gate_6_approval_satisfied_or_override_documented`, etc.).

**`above_authority` + target role DEPLOYED (Path A)**

- Auto-create **handoff artifact** (organizational handoff + task-mapping + escalation candidate bundle).
- Task **auto-routes** to target role's queue: staff → `senior_accountant` → `controller_helper` per 39D `escalationTargetRoleType`.
- Target role **prepares**; human still gates.
- **Does NOT** set `executable: true`.
- **Does NOT** set `noAutonomousEscalation: false` — artifact auto-created; **no autonomous AI-to-AI completion**.

**`above_authority` + target role NOT DEPLOYED (Path A — advise-to-deploy)**

- Emit **advise-to-deploy** message (structured notification artifact).
- Fires **only** when task genuinely exceeds authority **AND** target role genuinely absent per deployment index.
- **Never a nudge** — no message when task is in-lane or when target is deployed.
- Task stays with acting role in `flagged_for_review` / hold state until deploy or human disposition.

**Fraud / reasonableness flags (hard stop — all paths)**

- **Always** escalate to `human_controller`.
- **Never** auto-route to another AI role.
- **Never** convert to advise-to-deploy.
- Bypasses deployment index lookup for AI targets.

### Preserved boundaries (non-negotiable)

| Guardrail | Status in Path A |
|-----------|------------------|
| `executable: false` on all outputs | **Preserved** |
| `noAutonomousEscalation: true` on escalation packages | **Preserved** |
| `isRecommendationOnly: true` where Phase 40 requires it | **Preserved** for escalation *recommendations*; routing orchestrator creates **routing artifacts** separately (documented exception below) |
| Phase 40 non-goal: "Live execution of coordination decisions" | **Preserved** — routing = artifact + queue metadata, not runtime worker execution |
| Human final decision on material items | **Preserved** |
| Phase 42.5 control spine | **Out of scope** for this extension — platform isolation is a separate layer |

### Documented exception — "auto-route" framing

Path A **"auto-route"** means **deterministic creation of handoff + queue-assignment artifacts** and population of `approvalRoutingTargetRoleType` / `escalationTargetRoleType` on those artifacts.

It does **not** mean:

- An execution loop runs the target AI worker to completion
- `executable: false` is lifted anywhere in the namespace
- `noAutonomousEscalation` is relaxed on escalation packages
- Phase 40's "no live execution of coordination decisions" non-goal is violated

The target role **prepares** under existing human gates; completion remains human-gated.

### Path A vs Path B

| Capability | Path A (this extension) | Path B (deferred execution phase) |
|------------|-------------------------|-----------------------------------|
| Authority evaluation | Build now | Reuse evaluator |
| Handoff artifact on above-authority | Auto-create | Same + optional live dispatch |
| Queue routing metadata | Auto-populate target queue refs | Same + dequeue/enqueue runtime |
| In-lane work execution | Human-gated prepare only | Autonomous in-lane depth TBD |
| AI-to-AI completion | **Prohibited** | TBD — separate lock + guardrails |
| `executable: true` | **Never** | Separate phase decision |
| Advise-to-deploy | Build now | Unchanged |

### Verifier design (40V extension)

Add exported checks to `scripts/verify-si-organizational-operating-system.js`:

- Deployment index: fail-closed resolver returns `not_deployed` when record missing
- Evaluator: staff + forbidden family → `forbidden`; staff + manager reviewLevel → `above_authority`
- Evaluator: fraud flagged → `requires_human`, target `human_controller`
- Orchestrator: above_authority + deployed senior → routing package contains senior `taskQueueReferenceId`
- Orchestrator: above_authority + not deployed senior → `AdviseToDeployNotification`, **no** queue route
- Orchestrator: in_lane → no advise-to-deploy artifact
- All extension outputs: `executable: false` literal
- Escalation packages from fraud bridge: `noAutonomousEscalation: true` literal preserved

**Optional probe:** `scripts/probe-si-ad-hoc-authority-routing.js` — poison cases PC-AAR-01+ (forbidden in-lane claim, wrongful advise-to-deploy, fraud routed to AI, etc.).

### Extension exit criteria

- 40C-EXT deployment index builder + resolver export present
- 40E-EXT-A/B/C + 40F-EXT builders pass module audits
- 40V extension checks exit 0
- TypeScript clean under new namespaces
- Cross-isolation: no tenant A deployment record resolves in tenant B scope
- Documented proof: fraud flag never produces advise-to-deploy artifact (verifier + probe)

### Open dependency — 40C-EXT sequencing vs Phase 42.5

**40C-EXT is a hard prerequisite for production advise-to-deploy.**

Authority evaluator and orchestrator can be built with deployment index **stubbed to always `not_deployed`** in tests; production advise-to-deploy remains **blocked** until the index is populated.

**Sequencing question (founder decision):**

The deployment index inherits tenant scope via `companyId` + `customerIsolation` / `firmIsolation` / `clientIsolation` — the same isolation dimensions Phase 39/40 already use. Phase 42.5 will build and prove the **universal control spine** (isolation, RBAC, audit) that ultimately governs all panel data and tenant attributes.

Options:

| Option | Build 40C-EXT | Rationale |
|--------|---------------|-----------|
| **A — Build now with Phase 39/40 isolation fields** | Before or parallel with 40E-EXT | Unblocks Path A artifact routing; re-lock index against 42.5 D0 if spine semantics change scope keys |
| **B — Defer 40C-EXT until after Phase 42.5 D0** | After LOCK-42.5.3 | Index inherits proven spine isolation model once; advise-to-deploy waits |
| **C — Split** | Contract + fail-closed stub now; full resolver after 42.5 | Evaluator/orchestrator build proceeds; advise-to-deploy production gate tied to 42.5 |

**Redline expected:** which option binds before fold-in to `PHASE_40_PLANNING_DOCUMENT.md`.
